## 1. What we are checking

Given a Verus `exec` function `f` with an `ensures` clause, we want to
answer:

> **Is `f` deterministic with respect to its specification?**
> Equivalently: do all execution traces of `f` that satisfy the same
> `requires` end in observationally-equivalent post-states, where
> "observationally-equivalent" is the equivalence relation the ensures
> implicitly fixes?

This is *not* "is the implementation of `f` deterministic" (it usually
is ŌĆö Rust + Verus exec code is). The interesting question is whether
the **specification** uniquely characterises the post-state, or whether
the spec admits two implementations that disagree on something the
spec didn't pin down.

When the answer is "yes" we say `f` is **complete**: the spec is tight
enough to mechanically prove that two outputs agreeing on the spec
must agree everywhere we can observe. When the answer is "no" we say
`f` is **incomplete**: there exist two concrete post-states `(r1, r2)`
that both satisfy `ensures(f, args, ri)` yet differ ŌĆö these are
**witnesses of spec incompleteness**.

### 1.1 Why this matters

* **Spec quality auditing.** A spec that admits multiple posts is
  often under-constrained; the incompleteness witness localises the
  missing constraint to a particular field / bit / element.
* **Refinement proofs.** Downstream proofs (e.g. crash-consistency,
  state-machine refinements) implicitly assume their abstract spec
  pins the post-state. Spec incompleteness silently breaks them.
* **Auto-derived equalities.** If we want to mechanically derive
  "two executions agree iff their views agree", we need to know that
  the spec actually forces this.

### 1.2 The mechanical reduction

For each target `f(args) -> R` with `ensures Q(args, r)`, we synthesise
a Verus proof obligation:

```rust
proof fn det_<f>(<args>, r1: R, r2: R)
    requires Q(args, r1) && Q(args, r2),
    ensures  det_<f>_equal(r1, r2),
{
    // body filled in by the pipeline (schema search and/or LLM proof)
}
```

* `det_<f>_equal` is the **equal-fn** we synthesise (Stage 4): the
  equivalence relation that captures "two posts are the same as far
  as we care".
* If Verus accepts the body (z3 unsat under negation), `f` is
  **complete**.
* If z3 returns sat, we have a concrete `(r1, r2)` ŌĆö an
  **incompleteness witness**.
* If z3 returns unknown, we either escalate to Tier 3 (LLM proof) or
  bucket the target as **inconclusive**.

The rest of this document explains how every piece of that obligation
is produced and how the verdict is extracted.

## 2. The two design ideas the rest of the pipeline rests on

These come from the painful prior history documented in
`JOURNEY.md`. Skipping them means walking into the same traps.

### 2.1 Z3 is a pure sat/unsat oracle

We **never call `solver.model()`**. Counter-witnesses constructed from
Z3's `(get-model)` output are unreliable in two compounding ways:

1. Verus disables MBQI and uses incomplete quantifier patterns for
   arithmetic. On realistic goals z3 frequently answers `unknown`,
   and any subsequent `get-model` returns stale values from an
   earlier branch (or throws).
2. Even on sat, complex sorts come back as named universe elements
   (`Slab!val!7`, `Poly!val!36`). These are arbitrary names z3 is
   under no obligation to commit to. Walking view functions from them
   yields opaque "concrete" values that don't reduce.

**The pipeline therefore treats z3 only as a yes/no oracle.** Witnesses
are constructed from outside ŌĆö by *binary search itself*.

### 2.2 Search by guarded-template, not by re-invocation

The naive design re-runs Verus once per narrowing round (one Verus
invocation per `assume`). That cost ~2ŌĆō3 s per round ├Ś 20ŌĆō100 rounds
per function, and broke on big functions (`kernel::allocate` took
~13 min).

The current design:

1. **One Verus call per function** produces a single `.smt2` containing
   the proof goal *parameterised* by `(guard_i: bool, k_i: int, ...)`
   placeholders for every potential narrowing dimension.
2. The Python side loads that SMT-2 into an in-memory `z3.Solver`
   **once** and drives the search by toggling assumption literals
   (`solver.check(g_0==True, k_0==5, g_1==True, ...)`).
3. Each "round" becomes a sub-millisecond `check` call. Learned
   clauses persist across rounds (we never `pop`), giving another
   ~14├Ś speedup over push/pop.

**Witnesses are emitted by binary search itself**, not by reading a
model: when `solver.check(... assume_i ...)` returns `unsat`, that
assume is a *must-hold* condition, so we commit it to the witness. The
output is always a tree of Rust-level assumes that are concrete by
construction; they never bottom out at an abstract universe element.

These two invariants are why the pipeline is fast *and* produces
witnesses you can actually paste back into Verus.

## 3. Result vocabulary and the bucket funnel

The pipeline emits one of the following terminal labels per target:

| Bucket | Meaning |
|---|---|
| `complete` | z3 (alone) proved determinism. The spec uniquely pins the post-state. |
| `complete_llm` | z3 plus an LLM-authored proof annotation proved determinism. |
| `incomplete` | z3 produced a sat witness, **or** R0 returned unknown but the spec was detected as intentionally permissive (e.g. uses `|||`). The narrowed assumes are real incompleteness evidence. |
| `incomplete_permitted` | Subset of `incomplete` where the spec intentionally permits multiple posts (transparently flagged so paper claims can separate "spec bug" from "spec design"). |
| `ok_inconclusive` | R0 returned unknown and no escalation closed it. Not a verdict ŌĆö a "z3 couldn't decide" placeholder. |
| `verus_error` | Verus rejected the synthesised proof obligation itself (syntax, type error, deref strictness, etc.). Often a pipeline bug, sometimes a real source-code limitation. |
| `ok_unknown_kind` | Status is `ok` but `r0_z3` is an unexpected value. Legacy / forensic. |
| `no_artifact` / `no_smt2` / `search_error` / `exception` | Failure modes upstream of z3. |

`r0_z3` is the raw z3 verdict at R0 (no schema assumes). It is the
**ground truth** the bucket label is derived from; the bucket label
adds policy (LLM assistance, permitted-OR detection, manual real-sat
allowlist).

not as "deterministic by default".

("unknown = a third bucket alongside sat/unsat") leads to silently
under-reporting incompleteness. See `determinism-funnel-framework.md`
for the long version.

## Stage 0 ŌĆö corpus discovery

Two entry points, one for cargo workspaces and one for self-contained
single-file targets:

* **Workspace driver ŌĆö `spec_determinism/corpus/run_all.py`.** Used
  by the `nanvix` corpus. Reads a `corpus.toml` (loaded via
  `spec_determinism/config.py`) listing crates + target functions.
  Each entry maps to a `CrateConfig` with `proof` file (where we
  inject), `features`, `timeout`, `verify_module`, `use_build`, and
  optional `check_overrides` to rename the proof fn we synthesise.
* **Single-file driver ŌĆö `spec_determinism/corpus/verusage_run.py`.**
  Used by the seven verusage projects (atmosphere / ironkv / nrkernel
  / storage / anvil-library / memory-allocator / vest). Walks a
  directory tree of self-contained `.rs` files, picks any exec fn
  whose body has `ensures`, and runs the pipeline per `(file, fn)`.

The single-file driver does its discovery in `_discover_targets`:

1. Resolve each project root ŌåÆ `<root>/<project>/<subdir>`
   (typically `verified/`).
2. Recursively walk for `*.rs`, drop files with no `ensures` keyword.
3. Tokenise top-level `fn NAME(` decls via `discover_exec_fns`
   (Stage-1 extractor is called later to confirm and pull types).
4. Yield `(file, fn, artifact_key)` where `artifact_key` is
   `<project>__<rel/path/with/underscores>__<fn>` ŌĆö also the name of
   the per-target artifact directory.

Both drivers can apply project-level filters (`--only`,
`--skip-existing`, `--include-quarantined`, etc.) before handing off
to the per-target pipeline.

## Stage 1 ŌĆö `extract_spec`: source ŌåÆ `FunctionSpec`

Module: `spec_determinism/extract/extractor.py`.
Output dataclasses: `spec_determinism/extract/types.py`
(`TypeKind` / `TypeInfo` / `FieldInfo` / `VariantInfo` / `Param` /
`FunctionSpec`).

Tree-sitter-verus parses the Verus source. We walk the CST and emit
one `FunctionSpec` per target. The structure carries:

* `name`, `is_pub`, `self_type` (if it's a method), `trait_name`
  (if inside a `trait_item` ancestor).
* `params: list[Param]` ŌĆö each `Param` has `name`, `type: TypeInfo`,
  `is_ref` / `is_mut_ref` / `is_self`, and `destructure_ctor` (set
  to `"Tracked"` / `"Ghost"` when the parameter is destructured at
  the binding site so we can recover the inner type).
* `return_type: TypeInfo`.
* `requires_expressions` / `ensures_expressions` ŌĆö raw Rust source
  strings (cleaned of comments and leading whitespace). Kept as text
  because Verus parses them better than we can.
* `type_defs: dict[str, TypeInfo]` ŌĆö every named type reachable from
  parameter / return / field / variant types. The map is keyed by
  the bare head name (`HashMap`, not `HashMap<V>`) and contains a
  recursively-resolved tree.
* Attribute flags propagated through `extract/attrs.py`:
  `is_external_body`, `is_external_type_specification`,
  `is_ext_equal`, `cfg_feature_req`, `derives`. These ride on
  `TypeInfo.is_opaque` / `TypeInfo.is_ext_equal` and route the
  equal-fn generation later.
* `spec_view` per type: when the source has
  `impl T { spec fn view(self) -> ViewT }` (or an
  `impl View for T`), we populate `TypeInfo.spec_view = TypeInfo
  describing ViewT`. The codegen later uses this to emit
  `(r)@ == (r)@` instead of structural field-by-field equality.

### 1.1 Key sub-routines

* `_extract_params` ŌĆö handles tuple-struct destructure
  (`Tracked(g): Tracked<T>` records ctor + unwraps `T`),
  `&` / `&mut`, lifetimes, `mut`.
* `_resolve_types` ŌĆö a final post-pass that walks every reachable
  TypeInfo and re-runs attribute propagation, so generic
  instantiations (`AckState<Message>` constructed from
  `type_defs["AckState"]`) carry the same `is_opaque` /
  `is_ext_equal` / `spec_view` flags as their definitions.
* `_find_struct` / `_find_enum` ŌĆö populate `FieldInfo` /
  `VariantInfo` lists. Variants can be tuple-form (positional) or
  struct-form (named); `VariantInfo.struct_form` records which.

### 1.2 Failure modes

* **Macro-wrapped types.** Tree-sitter doesn't expand macros, so
  types defined inside e.g. `define_enum_and_derive_marshalable! {
  enum CSingleMessage { ... } }` are invisible. Falls through to
  Stage 2 (Tier 1.5).
* **Generic alias with no body.** `pub type AckList<T> = Seq<T>`
  comes through as `TypeKind.UNKNOWN`. Falls through to Tier 1.5.
* **Trait return type `Self`.** When the target is inside a trait
  default body, `Self` has no concrete resolution. We carry
  `trait_name` so `gen_det` can synthesise a placeholder
  `__DetSelf` generic.

### 1.3 Tests

`python -m spec_determinism.extract.extractor test` ŌĆö ~50 self-tests
covering: simple struct, tuple-struct, struct-form enum, generic
struct, attribute parsing, Ghost/Tracked destructure, alias
resolution, ext_equal propagation, external_body opacity.

## Stage 2 ŌĆö Tier 1.5 type completion (optional)

Module: `spec_determinism/llm_type/` (subpackage). Entry point:
`runner.complete_types`. Enabled by `--llm-type-completion` on the
single-file driver; optional `--llm-type-completion-pinned-autodetect`
picks up a project-pinned cache snapshot at
`verusage/cache_snapshots/<project>/`.

### 2.1 Why this stage exists

Stage 1 can fail on three patterns even when the source has the
information we need:

* **Generic-unresolved**: `HashMap<V>` is referenced but the
  tree-sitter lookup for the bare head `HashMap` misses the
  surrounding `pub struct HashMap<V> { ... }` because of impl/macro
  nesting.
* **Unknown kind**: `pub type AckList<T> = Seq<T>` ŌĆö alias with no
  struct/enum body.
* **Macro-wrapped**: type definitions hidden inside a macro
  invocation.

Without Tier 1.5, codegen falls back to a structural
`r1 == r2` over an opaque/missing type, which Verus then rejects
(field access on opaque) or which produces a wrong equal-fn (too
strong, masking the spec).

### 2.2 Workflow

```
complete_types(spec, project_root):
    while round < max_rounds:
        gaps = detect_gaps(spec, source)         # static, no LLM
        if not gaps: break
        for gap in gaps:
            patch = cache.get(gap.name)          # try live, then pinned
            if patch: apply_patch(spec, patch)
        remaining = gaps_still_open(spec)
        if not remaining: break
        # Single LLM call covers every remaining gap.
        prompt = build_prompt(remaining, source)
        raw = copilot_cli(prompt)
        patches = parse_llm_output(raw)
        for p in patches:
            v = run_gates(p)                     # V1 evidence, V2 type-str, V3 codegen smoke
            if v.passed: apply + cache.save
            else: cache.save(rejected=v.reasons)
        round += 1
```

### 2.3 Gates

| Gate | Check | Purpose |
|---|---|---|
| V1 evidence-existence | open `rel_path`, find `snippet` within ┬▒3 lines | Anti-hallucination |
| V2 type-str parses | `_parse_type_node` round-trips `type_str` via tree-sitter | LLM didn't invent a non-existent type syntax |
| V3 codegen smoke | `gen_det` produces a syntactically-valid det fn / equal_fn under the patched type_defs | The patch is composable with the rest of the pipeline |
| V4 (optional) verus compile | `verus --no-verify` over the produced det fn | Strong type-check; opt-in |

V4 is what the historical "Bug B compile probe" did. After empirical
measurement (every Bug B "win" had `shape_det=0` once the cache was
warm ŌĆö see `STATUS.md` and `determinism-funnel-framework.md` ┬¦3),
the probe was refactored out of the main path into the standalone
`scripts/validate_tier15_cache.py`. Use it before committing a new
pinned cache, not on every run.

### 2.4 Cache and pinning

* Live layer: `~/.cache/spec_determinism/type_completion/<project_hash>/`
  (per-user). Hash is sha256 of the project source corpus.
* Pinned layer: `verusage/cache_snapshots/<project>/` (in-repo).
  Pre-vetted snapshot. The runner reads `pinned_hash` and surfaces
  `pinned_hash_matches` in telemetry; reads from the pin are warm
  even when the source has drifted (with a flag to indicate drift).
* A/B reproducibility: every result carries
  `tier15.pinned_hash`, `tier15.current_source_hash`,
  `tier15.pinned_hash_matches`, `tier15.gap_sources` (per-name
  `"live" | "pinned" | "miss"`). Comparing two runs' telemetry tells
  you if the cache substrate changed.

### 2.5 LLM output schema

```jsonc
{ "type_patches": [
  { "name": "HashMap",
    "kind": "struct",                     // "struct" | "enum"
    "type_params": ["V"],
    "fields":   [ { "name": "m", "type_str": "..." } ],
    "variants": [ /* for enum */ ],
    "spec_view": { "type_str": "Map<AbstractEndPoint, V>" },
    "source_evidence": {
      "rel_path": "src/host_protocol_v.rs",
      "line": 164,
      "snippet": "pub uninterp spec fn view(self) -> Map<AbstractEndPoint, V>;"
    }
  }
] }
```

The applier (`apply.py`) converts each patch into a `TypeInfo` and
merges it into `spec.type_defs`. The applier uses
`dataclasses.replace` for generic instantiation shallow-copies (this
was historically a recurrent bug surface ŌĆö first `is_opaque`, then
`is_ext_equal` were dropped on the shallow copy).

### 2.6 Telemetry

`CompletionTelemetry` records: gap names, cache hit/miss totals,
hit source (`live` vs `pinned`), pinned-hash match flag, current
source hash, round count, accepted vs rejected patches, per-gate
rejection reasons.

## Stage 3 ŌĆö view resolution (L1/L2/L3/L4)

Module: `spec_determinism/view/` (subpackage). Entry point:
`registry.ViewRegistry.resolve(type_expr)` returning a
`Resolution(layer, view_type_text, view_expr, rationale, needs,
prelude_decl)`.

### 3.1 What "view resolution" means

Verus types frequently have a paired `View` impl: `Vec<T>@` has type
`Seq<T@>`, `Map<K, V>@` is `Map<K, V>` (identity), `Option<T>@` is
`Option<T@>`, user-defined `struct Page` may have an
`impl View for Page { type V = AbstractPage; ... }`.

The spec generally compares views (`r@ == r@`), but our default
equal-fn would compare values structurally. Without view-aware
codegen, we generate `r1 == r2` where the spec only inspects
`r1@ == r2@`, producing false-positive z3 sat witnesses.

The view resolver is the four-layer fallback that finds the right
`@` projection per type.

### 3.2 The four layers

| Layer | What | Source | Where |
|---|---|---|---|
| L1 prelude | Hard-coded Vec/Option/Map/Set/Box/Arc/Rc/&T/&mut T/Ghost/Tracked rules | `view/prelude.py` | n/a |
| L2 alias | Follow `pub type X = Y` and recurse | `extract/type_registry.py:alias_target_expr` | source |
| L3 impl scanner | Scan files for `impl View for X { ... }` blocks | `view/impl_scanner.py` | source |
| L4 LLM | Synthesise an `impl View for X` via Copilot when L1/L2/L3 miss | `view/llm.py` | LLM, cached |

The resolver returns the **first** matching layer's resolution; the
`uncovered` layer means even L4 didn't produce a usable answer.

### 3.3 Key prelude rules (L1)

* `Vec<T>` ŌåÆ view type `Seq<T@>`, view expr `({x})@`.
* `Option<T>` ŌåÆ view type `Option<T@>`, view expr `({x})@`.
* `Map<K, V>` ŌåÆ identity view. *Important*: Verus's
  `vstd::map::Map` is itself a spec primitive with no `.view()`
  method; emitting `(x)@` here causes E0599. The rule is
  `view_expr = ({x})` (identity), `view_type_text = Map<K, V@>`.
* `Set<K>` ŌåÆ identity view (same logic).
* `Box<T>` / `Arc<T>` / `Rc<T>` / `&T` / `&mut T` ŌåÆ transparent;
  view of the inner type.
* `Ghost<T>` / `Tracked<T>` ŌåÆ unwrap to inner T's view.
* `[T; N]` / `[T]` ŌåÆ `Seq<T@>`.

### 3.4 L3 ŌĆö impl scanner

`view/impl_scanner.py:scan_file(path) ŌåÆ ImplScan`. Tree-sitter walk
of `impl_item` nodes; we keep those whose trait is `View` and record
the impl head type, type parameters, bounds, and the rendered
`type V = ...;` body. The output is folded into a per-project
`ViewImpl` list and indexed by head name.

### 3.5 L4 ŌĆö LLM synthesiser

When L1/L2/L3 miss, `view/llm.py` builds a prompt with the type
definition + already-resolved views of its fields + relevant
ensures snippets, asks the LLM to synthesise a
`pub closed spec fn view_<T>(...) -> ...` plus a thin
`impl View for T` wrapper. The result is:

1. Validated by `verus --parse-only` (syntactic check).
2. Critiqued via `view/critic.py` ŌĆö a separate LLM pass that
   applies semantic lints (`M1`..`M4` and others; see
   `docs/critic-criteria.md`).
3. Static-linted by `check_view_body_uses_self` and the M1/M2/M3
   detectors.

Accepted entries are cached at
`results-verusage/view_registry/<project>/<type>.json`:

```jsonc
{ "type": "Page",
  "source_hash": "<sha>",
  "view_source": "L1-prelude" | "L2-alias" | "L3-impl" | "L4-llm",
  "verus_source": "pub spec type ViewPage = ...;\npub closed spec fn view_Page(...) -> ...",
  "depends_on_views_of": ["PagePtr", "PageState", ...],
  "critic_verdict": "accept" | "revise" | "reject",
  "critic_issues": [...] }
```

Rejected entries are written to a sibling `_rejected.jsonl` and never
fed back into the registry. Quarantined entries (manually disabled
after a regression diagnosis) get a `.json.quarantine` suffix and
are skipped on load unless `--include-quarantined`.

### 3.6 Wiring into codegen

`codegen/gen_det.py:build_equal_expr` consults `ViewRegistry.resolve`
for every struct / enum / container type it encounters. The returned
`view_expr` is the function used to render the equality body:
`view_expr("r1") + " == " + view_expr("r2")`. For types that
require an L4 `prelude_decl`, that block is emitted right before
the synthesised equal-fn in the injected file so Verus can
type-check it.

### 3.7 Per-type compatibility checks

After L4 view emission worked, two further checks were needed to
avoid false-positive E0277s:

* **Shim trait-bound compatibility (`_shim_bounds_satisfied`).** If
  the L4 shim is `impl<K: KeyTrait + VerusClone + View> View for X<K>`
  but the caller's synthesised equal-fn only has
  `<K: KeyTrait + VerusClone>`, the shim is rejected ŌĆö fall back to
  field-level instead of emitting an unverifiable view eq.
* **Opaque Ōł¦ has-view fallthrough.** When a STRUCT has both
  `is_opaque=True` (via `external_body`) and `spec_view=Some(...)`,
  emit `(r1)@ == (r2)@` (use the view), not `r1 == r2` (opaque
  short-circuit). The view is the right abstraction; bare structural
  equality on opaque types is too strong because the ensures only
  pin `r@`.

## Stage 4 ŌĆö equal-fn policy and synthesis

Module: `spec_determinism/codegen/equal_policy.py`. Data class:
`EqualPolicy`. Default: `default_policy()` = `errs_equivalent=True`.

The equal-fn decides what counts as "two outputs are the same as far
as we care". The default policy is "structural equality everywhere
except all-Errs-are-equal (the spec rarely pins down which error
code is returned)". The policy is consulted by
`build_equal_expr` recursively, and supports five knobs:

* `errs_equivalent: bool` ŌĆö collapse all `Err _` to one.
* `opaque_ok: bool` ŌĆö collapse all `Ok _` to one (rare;
  allocator-style indices).
* `compare_raw_pointers: bool` ŌĆö by default `*mut T` / `*const T`
  are treated as opaque (allocator-nondeterministic).
* `ignore_fields: set[str]` ŌĆö drop named fields from the comparison
  (e.g. internal caches).
* `opaque_types: set[str]` ŌĆö type names whose instances all compare
  equal.
* `custom_body: Optional[str]` ŌĆö verbatim Verus code that replaces
  the synthesised body. Human-/LLM-written, highest priority.

The `is_default()` check is used by an optional LLM hook to decide
whether to overwrite the policy (only if untouched).

### 4.1 `build_equal_expr` ŌĆö the body synthesiser

Located in `gen_det.py`. Recursive descent over `TypeInfo`:

1. **Policy short-circuit.** `policy.opaque_types` / `opaque_ok` /
   `ignore_fields` hit first.
2. **ext_equal short-circuit.** If the type carries
   `#[verifier::ext_equal]` (or `#[verifier(ext_equal)]`), emit
   `(lhs) =~= (rhs)`. Sound because `=~=` is at-least-as-strong as
   structural field-wise comparison on ext_equal types.
3. **View short-circuit.** If `ty.spec_view is Some(v)` (or the
   resolver yields a non-uncovered Resolution), emit
   `({lhs})@ == ({rhs})@`. This skips field-by-field on view types.
4. **Opaque short-circuit.** If `ty.is_opaque` (external_body) and
   no view is available, emit `(lhs) == (rhs)` (Verus's structural
   `==` on opaque types is the strongest we can do).
5. **STRUCT branch.** Recurse into each `FieldInfo`, AND together.
   Honour `ignore_fields`.
6. **ENUM branch (after C-like discriminant collapse).** Emit a
   `match` over both sides, AND together per variant. Tuple-form
   variants use `r->Variant_0`; struct-form uses `r->fieldname`.
   When the same field name appears in multiple struct-form
   variants of the same enum, fall back to `lhs == rhs` under the
   discriminator guard `lhs is V` (the `r->fname` accessor is
   ambiguous to Verus's `arrow_f` resolver).
7. **Container branches.** Vec/Seq/Map/Set/Option/Result follow
   prelude rules.
8. **Tuple branch.** Per-component equality, applying the same
   recursion to each component.

### 4.2 ENUM view short-circuit (Task B, 2026-05-20)

When the enum has `spec_view` set (e.g. CSingleMessage with
`view ŌåÆ SingleMessage<Message>`), short-circuit straight to
`(r1)@ == (r2)@` instead of per-variant computation. Mirrors the
struct branch. This is what closed the IronKV
`clone_up_to_view` / `send_single_cmessage` cluster (8 unsat wins
on rerun9 vs rerun8).

### 4.3 Tuple per-component view emission

For multi-output `(bool, Seq<Packet>, ...)`, each component is
recursed independently and runs through `build_equal_expr` with
its own resolver lookup. The tuple equality is `c0_eq && c1_eq && ...`.
A common slip is "tuple-equal forgets to view a component"; the
codegen as of 2026-05-20 emits per-component view correctly
(verified on `send_packet*` / `receive_packet_next` cluster).

## Stage 5 ŌĆö `gen_det`: synthesising the proof obligation

Module: `spec_determinism/codegen/gen_det.py`. Entry point:
`build_det_check_spec(spec, source=..., policy=...) -> DetCheckSpec`.

The `DetCheckSpec` is the structured handoff to the schema-search
stages. It carries:

* `spec: FunctionSpec` (from Stage 1, possibly Tier-1.5-augmented).
* `det_fn_name: str` ŌĆö typically `"det_<fn>"`. Cargo-corpus targets
  may override via `check_overrides`.
* `det_fn_template: str` ŌĆö the Rust source of the proof fn with
  `{ASSUMES}` placeholder where the rendered guard/k-conditional
  body will be spliced.
* `equal_fn_name: str` and `equal_fn_def: str` ŌĆö the synthesised
  `spec fn det_<fn>_equal(...) -> bool` plus its Rust source.
* `symbols: list[Symbol]` ŌĆö every named SMT-level symbol the search
  tracks. Each `Symbol` is an `(name, type: TypeInfo, kind)` triple
  used to drive narrow strategies and to enumerate schemas.
* `policy: EqualPolicy` ŌĆö for record-keeping / re-rendering.
* `opened_closed_specs: list[str]` ŌĆö the list of qualified spec-fn
  names we opened (see ┬¦5.3 below).
* `tier15` telemetry (if Tier 1.5 ran).

### 5.1 Template construction (`_build_template`)

1. Build the proof-fn signature:
   `proof fn det_<fn><GENERICS>(<params>, ..., g_neq_tuple: bool,
   g_<schema_i>: bool, ..., k_<schema_i>_<j>: int, ...)`.
   Generics are stripped via `_prune_generics` to drop phantoms;
   `Self` is replaced by either `self_type` or a synthetic
   `__DetSelf` placeholder when the fn is in a trait body.
2. Generate the `requires` clause: substitute every reachable
   ensures expression of `f` with `r1` / `r2` substituted for the
   return symbol. Two clauses: `<ensures>(args, r1) &&
   <ensures>(args, r2)`.
3. Generate the `ensures`: `det_<fn>_equal(r1, r2)`. Negated form
   for the SMT obligation is emitted via `if g_neq_tuple {
   assume(!det_<fn>_equal(r1, r2)); }` inside the body.
4. Generate the body: a `reveal(<closed_spec_fn>);` line per opened
   spec fn (see ┬¦5.3) followed by the `{ASSUMES}` placeholder.

### 5.2 Substitutions (`_substitute_input` / `_substitute_ensures` /
`_substitute_self_type`)

These do the careful textual surgery the rest of the pipeline
depends on:

* `_substitute_input` ŌĆö when the source ensures mentions `*p` for a
  by-reference param, the synthesised proof fn passes `p` by value
  (so `*p` becomes `p`). For `&mut T` params, `*old(self)` / `*self`
  are stripped to `pre_self_` / `post_self_`. The function also
  rewrites `final(p)` / `final(self)` to the post-state variable
  the pipeline already binds.
* `_substitute_ensures` ŌĆö runs AST-aware identifier rename on every
  ensures so that match-arm-bound identifiers don't collide with
  pipeline-injected names.
* `_substitute_self_type` ŌĆö when `Self::method(...)` appears in
  ensures, rewrite to `Base::<Args>::method(...)` (turbofish for
  path-expression context); `Self<Args>` in type-expression context
  becomes `Base<Args>`. This is what made `HashMap<V>::get_spec`
  resolve correctly.

### 5.3 Opening closed spec fns (`659c9bdc`)

When the target's ensures transitively reaches a `closed spec fn`
defined in the same module, z3 cannot inspect the body ŌĆö the closed
modifier hides it. We don't want to flip `closed ŌåÆ open` (that
forces body well-formedness at every external call site, which
breaks if the body references pkg-private fields). Instead:

1. `classify.reachable_spec_fns(spec, source)` finds every spec fn
   the ensures transitively calls.
2. `classify.closed_spec_fn_qualified_names(...)` picks the closed
   ones.
3. `classify.rewrite_closed_to_opaque(source, names)` injects
   `#[verifier::opaque]` on each (Verus accepts the combo
   `#[verifier::opaque] pub closed spec fn`; body stays
   closed-by-default but is now revealable).
4. `gen_det._build_template` emits `reveal(<qualified_name>);`
   inside the proof body before the assumes.

Skip rules: decls already marked `#[verifier::external_body]` or
already opaque-tagged, and bodyless trait-method forward decls.

### 5.4 Test coverage

`python -m spec_determinism.codegen.gen_det test` ŌĆö ~70 self-tests
covering: STRUCT/ENUM branches under every combination of opaque,
ext_equal, spec_view, ignore_fields; tuple-component view emission;
Self / trait substitution; ambiguous struct-form variant fallback;
`_substitute_input` self deref strip; turbofish substitution; opaque
spec-view fallthrough; reveal injection.

## Stage 6 ŌĆö schema enumeration and guarded-template rendering

Module: `spec_determinism/schema_search/schemas.py`. Functions:
`enumerate_schemas(det_spec) ŌåÆ list[SchemaBinding]`,
`render_guarded_template(det_spec, schemas) ŌåÆ str`,
`translate_assume(assume, schemas, equal_fn_name) ŌåÆ Optional[(id, k_bindings)]`.

### 6.1 What a schema is

A **schema** is one pre-declared narrowing dimension. It carries a
`(guard: bool, k: int/...)` parameter pair on the proof fn. The
guard activates the assume body; the k-param (if any) makes the
schema polymorphic in its concrete value so the same schema covers
every bisection round.

There are ~12 schema kinds (`SchemaKind` enum; full table in
Appendix A):

| Kind | Shape of assume |
|---|---|
| `SCALAR_EQ` | `var == k` |
| `SCALAR_RANGE` | `k_lo <= var && var <= k_hi` |
| `BOOL_EQ` | `var == k_bool` |
| `STR_EQ` | `var@ == "literal"@` (uses `@` to compare view) |
| `VARIANT_IS` | `var is <Variant>` (Option/Result/enum) |
| `ENUM_DISC_EQ` | C-like enum: `var as int == k` |
| `SET_EMPTY` | `var =~= Set::empty()` |
| `SET_LEN_GT` | `var.len() > 0` |
| `SET_LEN_EQ` | `var.len() == k` |
| `SET_LEN_RANGE` | bisect on set len |
| `SET_CONTAINS` | `var.contains(k)` |
| `SEQ_LEN_EQ` / `SEQ_LEN_RANGE` | analogous to set |
| `NOT_EQUAL_FN` | `!det_<fn>_equal(r1, r2)` ŌĆö the goal itself |

### 6.2 Enumeration recursion

`enumerate_schemas` walks every tracked symbol from `det_spec`,
recursively emitting schemas for nested types. For an
`r1: Result<Option<Vec<Page>>, Error>` we emit:

* `VARIANT_IS r1 == Ok`, `VARIANT_IS r1 == Err`,
* under `r1 is Ok`: `VARIANT_IS r1->Ok_0 == Some`,
  `VARIANT_IS r1->Ok_0 == None`,
* under `r1->Ok_0 is Some`: `SEQ_LEN_RANGE r1->Ok_0->Some_0.len()`,
  plus per-element schemas up to `MAX_SEQ_LEN = 8`,
* under `r1 is Err`: per-field `Error` schemas,
* a terminal `NOT_EQUAL_FN`.

Each schema records its **parent guard chain**, so child schemas
only "fire" when the parent variant guard is on. This avoids
emitting contradictory dimensions (`r1 is Ok` AND `r1 is Err`).
Typical enumeration yields 25ŌĆō520 schemas per fn (median ~100).

### 6.3 Template rendering (`render_guarded_template`)

Splices the schemas into `det_spec.det_fn_template`'s `{ASSUMES}`
placeholder, producing the body:

```rust
if g_<schema_0> { assume(<rendered with k_<schema_0>_*>); }
if g_<schema_1> { assume(<...>); }
...
if g_neq_tuple { assume(!det_<fn>_equal(r1, r2)); }
```

Each schema's body re-asserts its parent chain first (Verus
needs the variant assertion before it allows the projection
expression). These re-asserts are idempotent so cost nothing if
the parent guard is already on.

### 6.4 `translate_assume` ŌĆö the dispatcher

For each Rust-level Assume we want to push, this looks up the
schema set and dispatches via `pred.match_and_bind(schema)` ŌĆö
the structured matcher each `AssumePred` carries. Returns
`(schema_id, k_bindings)` on success; `None` if no schema covers
this pred (caller falls through as `pass_untranslatable`).

This is the single home for pred Ōåö schema matching: adding a new
predicate kind is a one-place edit (`predicates.py`), adding a
new schema kind is at most three places (the pred's matcher,
`enumerate_schemas`, `_render_*`).

## Stage 7 ŌĆö Verus invocation and SMT-2 capture

Module: `spec_determinism/verus/verify.py` (workspace) and
`spec_determinism/verus/single_file.py` (verusage).

### 7.1 Workspace path

`inject_proof_fn(cfg.proof, code)` injects the rendered proof
function into the target crate's proof file (preserving the original
contents for restore). `run_cargo_verus(...)` invokes
`cargo +<toolchain> verus verify` with:

* `--target <target>` (e.g. nanvix's `x86_64-unknown-none`),
* `--features <list>`,
* `--verify-function det_<fn>`,
* `--verify-module ...` (if scoped),
* `--log-all --log-dir <tmp>` to capture the per-query SMT-2.

Returns a `VerifyResult` dict with `stdout`, `stderr`,
`returncode`, and the path to the dumped SMT-2 (which lives under
`/tmp/aprime_<crate>_<fn>_XXXX/`). The injected code is reverted
via `restore_file` in a `finally` clause.

### 7.2 Single-file path

`single_file.run_single_file(file, fn, ...)` does:

1. Read and reparse the file via `extract_spec`.
2. Optionally run Tier 1.5 (Stage 2).
3. Run `build_det_check_spec` (Stage 5).
4. Run `enumerate_schemas` + `render_guarded_template` (Stage 6).
5. **Source rewriting** (`_inject_into_source`):
   * `_rewrite_self_eq_old_self` / `_rewrite_ref_eq_ref` ŌĆö legacy
     deref-strictness fixups (replace `self == old(self)` ŌåÆ deref
     form), needed because some source files predate current Verus.
   * `_rewrite_mut_self_in_ensures` ŌĆö per-fn, per-clause `&mut`
     param rewriter (bare `p` ŌåÆ `final(p)` / `*final(p)`
     depending on context).
   * `_synthesize_view_trait_impls` ŌĆö discover inherent `spec fn
     view` and synthesise `impl View for X` with `external_body`,
     so other parts of the file that expect the `View` trait
     compile.
   * `rewrite_closed_to_opaque` ŌĆö see ┬¦5.3.
   * Splice the rendered det_fn + equal_fn at the end of the
     `verus! { ... }` block.
6. Write the modified file to a fresh temp path; shell out to
   `verus --no-color --emit=metadata --verify-function det_<fn>
   --rlimit 60 ...`.
7. Locate the produced `*.smt2` log file; parse it in Stage 8.

The rewrites are textual (regex + a small tree-sitter-aided walker)
and are kept conservative: every rewriter has self-tests pinning its
boundaries (a frequent regression class).

### 7.3 What gets dumped

The `.smt2` is the full Verus ŌåÆ z3 translation for the proof
fn. It contains:

* `(declare-sort ...)` and `(declare-fun ...)` for every reachable
  Verus type and function.
* Function-axioms (e.g. `(=> (fuel_bool det_X_equal.) (forall r1 r2
  (= (det_X_equal r1 r2) ...)))`).
* The proof obligation inside one `(push) ... (pop)` block per
  function, terminated by `(check-sat) (get-model)`.

We do NOT consume the `(get-model)` output. We split out everything
**outside** the push/pop blocks (which is the self-contained global
context: declarations + every Function-Axioms block) and concatenate
that as the prelude. The Stage-8 driver pushes the proof goal on
top of this prelude in a fresh z3.Solver.

### 7.4 Failure modes

If Verus rejects the synthesised file outright (E0599, E0277,
"postcondition not satisfied" on `__DetSelf` placeholders, etc.),
the result is bucketed as **`verus_error`** with a `stderr_tail`
truncated to 2KB. These are usually pipeline bugs (codegen produced
something Verus doesn't accept) ŌĆö the recurring categories are
tracked in `STATUS.md`'s "Known issue inventory" / `ISSUES.md`.

If Verus accepts the proof template (no errors), the per-target
control flow continues into Stage 8 regardless of whether Verus
itself says "verified" or "errors: 1 (postcondition not
satisfied)" ŌĆö the latter is the **expected** path because z3 then
gets a chance to find a sat witness for `!det_<fn>_equal(r1, r2)`.

## Stage 8 ŌĆö `build_schema_ctx`: load SMT-2 into z3-py

Module: `spec_determinism/schema_search/search.py`. Function:
`build_schema_ctx(smt2_path, fn_name, schemas, crate_name) ŌåÆ
SchemaCtx`.

### 8.1 Splitting the SMT-2

`_split_global_and_blocks(text)` walks `(push)` / `(pop)` tokens at
line granularity and returns the global text outside any push/pop.
This is the declarations + axioms; the only thing we strip is the
per-function check-sat blocks (we replace them with our own search).

An earlier version truncated at the first `(push)`, which dropped
the function-axioms blocks Verus emits **between** queries, and
let z3 vacuously satisfy `!det_X_equal(r1, r2)` (no axiom ŌåÆ
`det_X_equal` is uninterpreted ŌåÆ trivially refutable). The fix is
why we walk every push/pop pair.

### 8.2 Loading into z3.Solver

We construct a fresh `z3.Context` and `z3.Solver`, load the global
text via `solver.from_string(prelude)`, then assert the proof
obligation we care about (`assert (not <det_check_goal>)`).

### 8.3 Resolving guard/k constants

Each `SchemaBinding` has a `guard_name` (`g_<schema_i>`) and zero or
more `k_param` names (`k_<schema_i>_<j>`). Verus mangles these into
SMT-level names (`g_schema_i.` etc.), but the mangling is
deterministic and the names round-trip. We walk the parsed AST,
match by name pattern, and bind each guard / k to a `z3.BoolRef` /
`z3.IntRef` constant. The result lives on `SchemaCtx.guard_consts`
/ `SchemaCtx.k_consts`.

### 8.4 Performance

Per `kernel::allocate` (the worst case in our corpus, 5 MB SMT-2),
this loads in ~100ŌĆō130 ms. Median load time is ~40ŌĆō70 ms. The cost
is paid **once** per function ŌĆö every subsequent narrowing round
calls `solver.check(*literals)` on the same solver instance, in the
sub-millisecond range when the SMT formula isn't dominated by
quantifiers.

## Stage 9 ŌĆö R0 baseline check and bucket classification

Module: `spec_determinism/schema_search/search.py`. Function:
`run_schema_search(det_spec, schema_ctx) ŌåÆ Witness`. The R0 check
happens at the very top:

```python
r0 = solver.check(g_neq_tuple == True)   # all other guards off
if r0 == z3.unsat:   r0_z3 = "unsat"     # complete
elif r0 == z3.sat:   r0_z3 = "sat"       # incomplete witness
else:                r0_z3 = "unknown"   # inconclusive
```

`r0_z3` is persisted on the result and is the ground truth used by
`classify_ok` (Stage 12). Stage 10 (narrowing) runs regardless of
r0_z3, but its meaning depends on the baseline:

* `r0 == unsat` ŌåÆ already deterministic; narrowing committed
  assumes are *unused refinement attempts*. Bucket: `complete`.
* `r0 == sat` ŌåÆ real incompleteness; narrowing tells us *which
  fields* are responsible. Bucket: `incomplete`.
* `r0 == unknown` ŌåÆ z3 surrendered. Narrowed assumes are built on
  top of an undecided baseline and are unreliable. Bucket:
  `ok_inconclusive` unless Stage 11 (LLM proof) flips it.

## Stage 10 ŌĆö schema-driven witness narrowing (`AssumeTree` + `narrow_*`)

Module: `spec_determinism/extract/narrow.py`. Driver:
`SchemaSearchContext` in `schema_search/search.py`.

### 10.1 AssumeTree

`AssumeNode` is the search tree. Each node holds one `Assume` (a
candidate narrowing) and a dict of children keyed by string
(field name, variant payload index, `@view`, etc.). The tree shape
follows the type structure: `Result<Option<Seq<T>>, E>` makes a
chain `root ŌåÆ Ok_0 ŌåÆ Some_0 ŌåÆ Seq[i]` for `i in 0..MAX_SEQ_LEN`,
plus `root ŌåÆ Err_0` with its own subtree.

Refinement (e.g. `[0,8] ŌåÆ [3,4] ŌåÆ 3`) replaces the parent node's
`assume` in place; sibling refinements accumulate.

### 10.2 The narrow strategies

Registered per `TypeKind` via decorator:

| TypeKind | Strategy | What it does |
|---|---|---|
| INT / USIZE / ... | `narrow_integer` ŌåÆ `_bisect_range` | Classic lo/hi bisect on `SCALAR_RANGE` then commit `SCALAR_EQ` |
| BOOL | `narrow_bool` | Two `BoolPred`s |
| STR | `narrow_str` | Three literal candidates |
| SET | `narrow_set` | Empty? ŌåÆ len bisect ŌåÆ contains-enum |
| SEQ | `narrow_seq` | Length bisect + per-element recursion |
| OPTION | `narrow_option` | `is Some` / `is None`, then recurse into `Some_0` |
| RESULT | `narrow_result` | `is Ok` / `is Err`, then recurse |
| ENUM | `narrow_enum` | Per-variant `VariantIsPred` + recurse payload |
| STRUCT | `narrow_struct` | Recurse into each field |
| GHOST / TRACKED | unwrap to inner |
| UNIT | no-op |
| OPAQUE | no-op (cannot inspect) |

Every strategy calls `ctx.test_and_set(node, Assume.from_pred(...))`
and is agnostic to the concrete driver.

### 10.3 `test_and_set` semantics

```python
def test_and_set(self, node, assume, phase=""):
    node.assume = assume   # tentative
    assumes = self.tree.collect_assumes()
    literals = self._assumes_to_z3(assumes)   # via translate_assume
    if any pred untranslatable:
        node.assume = old; trace("pass_untranslatable"); return False
    r = self.solver.check(*literals)
    if r == z3.unsat:
        node.assume = old   # not needed; revert
        trace("pass"); return False
    else:
        # keep the assume ŌĆö it's a fail/sat/unknown, more narrowing needed
        trace("fail"); return True
```

The semantics are inverted relative to "does this assume hold". A
`pass` (unsat under this assume) means the assume is a *must-hold*
condition ŌĆö we *commit* it to the witness as a fact, and move on
without further narrowing this dimension. A `fail` means the assume
isn't strong enough to force determinism; keep narrowing.

This is what "binary search itself emits the witness" means: every
committed assume is the result of a successful unsat call.
Witnesses are concrete by construction ŌĆö we never stop a dimension
on an abstract universe element.

### 10.4 `_add_distinctness_witnesses`

After all narrowing, the driver tries one final
`assume(!det_<fn>_equal(r1, ..., r2, ...))`. If this is `pass`
(solver proves equality *must* hold given the accumulated
narrowing), nothing is added. If it's `fail`, the pred
`NotEqualFnPred(call=...)` joins the witness as a strong
distinctness assertion: "we narrowed everything we could, and the
two posts are still observably distinct".

### 10.5 Untranslatable assumes

`translate_assume` may return `None` when no schema covers a pred
(e.g. an exotic Set operation we don't emit schemas for). The
search treats this as `pass_untranslatable` and continues ŌĆö the
witness is still sound (the missing assume can only weaken it).

### 10.6 Output: `Witness`

```python
@dataclass
class Witness:
    assumes: list[Assume]    # the committed tree, flattened
    trace: list[dict]        # per-round telemetry
    r0_z3: str               # baseline z3 verdict
    n_rounds: int
    n_pass: int
    n_fail: int
    n_pass_untranslatable: int
```

Each `Assume` has `var`, `pred`, `description` (human-readable),
and its rendered Rust expression (derived from `pred.to_rust()`).
The witness is what gets dumped under `assumes` in the per-target
result JSON.

## Stage 11 ŌĆö Tier 3 LLM proof annotation (optional)

Module: `spec_determinism/llm_proof/` (subpackage). Entry point:
`run_llm_proof_loop`. Enabled by `--use-llm-proof` (or env
`SPEC_DET_LLM_PROOF=1`) on the single-file driver. Triggered only
when Stage 9 returned `r0_z3 == "unknown"`.

### 11.1 Why this stage exists

After Stage 10 finishes, some targets are still `r0_z3 == unknown`
not because the spec is incomplete but because z3 hit a quantifier
wall (Verus disables MBQI; arithmetic-heavy ensures with `forall`
clauses time out). The way to break the wall is to write Verus
proof annotations (`assert ... by`, `reveal`, `broadcast use`,
trigger annotations, helper lemmas) inside the proof body.

This is **proof completion**, not witness finding. The output is
labelled `complete_llm` so paper claims can distinguish baseline
z3 proofs from LLM-assisted ones.

### 11.2 Workflow per attempt

```
for attempt in range(max_attempts):
    prompt = build_proof_prompt(det_spec, prior_failure)
    raw = copilot_cli(prompt, timeout=600s)
    parsed = parse_proof_response(raw)       # extracts proof_block + helper_lemmas
    violations = scan_proof_block(parsed.proof_block)
    if violations:
        attempt.status = "sandbox_reject"
        prior_failure = format_violations(violations)
        continue
    # Re-render the det fn with proof_block appended.
    file = render(det_spec, proof_block=parsed.proof_block,
                  helper_lemmas=parsed.helper_lemmas)
    rc, stderr = verus(file)
    if rc == 0:
        cache.save(verus_pass, proof_block, helper_lemmas, shape_key)
        return success                       # ŌåÆ complete_llm
    else:
        attempt.status = "verus_reject"
        prior_failure = stderr[-2000:]
return failure                               # ŌåÆ still ok_inconclusive
```

### 11.3 The sandbox

`sandbox.scan_proof_block` walks the proof block AST and forbids
anything that could trivially close the proof unsoundly:

* `assume(...)`, `admit(...)`, `assume_specification`,
* `#[verifier::external_body]`, `#[verifier(external_body)]`,
* any non-`lemma_*` `proof fn` declaration (we only allow helper
  lemmas, which still have to be proved),
* `spec fn` / `exec fn` declarations,
* `struct` / `enum` / `trait` / `impl` blocks.

`sandbox.scan_helper_lemmas` is the stricter variant applied to the
optional ```` ```verus_lemmas ```` block: only `proof fn lemma_*`
decls are allowed.

This is what keeps the LLM's output *trustworthy*: even if Copilot
produces something silly, the only path to acceptance is Verus
agreeing the body type-checks and proves the obligation.

### 11.4 Patterns the prompt encourages

* **Pattern A ŌĆö helper proof fn declarations.** The model may
  declare module-scope `proof fn lemma_<name>` and call them from
  the proof body. The runner splices them next to the det fn (with
  a marker comment).
* **Pattern B ŌĆö `assert ... by { ... }`** blocks for local
  proof obligations.
* **Pattern C ŌĆö relational lemma hints.** When the source has spec
  fns of shape `step(pre, post, in, out) -> bool`, the prompt
  appends "suggested helper lemmas" sections pointing at each.

### 11.5 Cache

Two-level: by exact `(det_fn_text + source_hash)` and by
**shape key** (whitespace-normalised, comments stripped, generic
args collapsed). Shape-fallback lets a successful proof on
`delegation_map.insert` re-apply to the structurally-identical
`delegation_map.set` without re-LLMing. On shape hit, Verus
re-verifies and the entry is promoted into the current
artifact_key so the direct path catches it next time.

Cache schema (v2): `artifact_key`, `shape_key`, `proof_block`,
`helper_lemmas`, `verus_outcome Ōłł {verus_pass, verus_fail,
sandbox_reject, llm_failure}`, `timestamp`.

### 11.6 Telemetry

Per attempt: iteration number, llm_ms, verus_ms, status, stderr
tail. Per artifact: total wall, `llm_assisted: bool` (set on
`verus_pass`), final r0_z3 (overwritten to `"unsat"` on
`complete_llm`).

### 11.7 Empirical yield (IronKV rerun10, 2026-05-21)

Targeted Tier 3 on the 42 strict-unknown IronKV cases:
**4/42 = 9.5% pass rate**, dominated by simple uniqueness /
sorted-insert / delegation lookup lemmas. Wall: ~13k seconds for
the parallel run. Most failures hit the 600s per-Copilot-call cap
in both attempts; large-quantifier cases remain hard.

## Stage 12 ŌĆö permitted-incompleteness detection

Module: `spec_determinism/classify.py`. Functions:
`ensures_uses_permissive_or(spec, source)`,
`is_real_sat_manual_function(spec)`, `classify_ok(result)`.

### 12.1 Why this stage exists

Some Verus ensures intentionally permit multiple post-states:

* **Structural permissive OR.** Ensures uses `|||` directly or via
  a transitively-referenced `[open|closed] spec fn` body
  (e.g. IronKV's `next_delegate_postconditions`). The spec is
  saying "either of these posts is OK".
* **Manual real-sat.** A small curated list of spec fns whose
  ensures permit multiple posts by leaving return components
  unconstrained (no `|||` to detect structurally).

These are by-design "non-deterministic" and should not be reported
as accidental incompleteness.

### 12.2 Detection

* `ensures_uses_permissive_or(spec, source)` walks each ensures
  expression and looks for `|||` directly, then resolves any spec
  fn calls and walks their bodies up to depth 4.
* `REAL_SAT_MANUAL_FNS` is a hard-coded set keyed by qualified fn
  name; `is_real_sat_manual_function(spec)` checks membership.

### 12.3 Bucket promotion in `classify_ok`

```python
def classify_ok(result):
    r0 = result.get("r0_z3")
    permitted = result.get("permitted", False)
    llm = result.get("llm_assisted", False)
    if r0 == "unsat":   return BUCKET_COMPLETE_LLM if llm else BUCKET_COMPLETE
    if r0 == "sat":     return BUCKET_INCOMPLETE
    if r0 == "unknown":
        if permitted:   return BUCKET_INCOMPLETE   # promote!
        else:           return BUCKET_INCONCLUSIVE
    return BUCKET_UNKNOWN_KIND
```

The promotion `unknown + permitted ŌåÆ incomplete` is the right call
because we *know* the spec admits multiple posts; z3's failure to
decide isn't the issue. The result carries `permitted_reason Ōłł
{"permissive_or", "spec_underconstrained_manual"}` for the
renderer to surface.

`unsat + permitted` stays `complete` ŌĆö the spec admits multiple
posts but z3 proved this particular trace is uniquely determined
under the chosen equal-fn (e.g. all permitted variants are
view-equal).

## Cross-cutting concerns

### Quarantines

When a view-registry L4 entry or a Tier 1.5 cache entry causes
downstream regressions, we rename the JSON to add a
`.quarantine` (or `.quarantine.<reason>`) suffix. The loader skips
suffixed files unless `--include-quarantined`. Quarantine is the
escape valve for "this LLM output is wrong but we don't want to
delete the evidence". Current quarantine inventories live in
`results-verusage/view_registry/<project>/`.

### LLM determinism / cache stability

LLM-using stages (3 L4, 2 Tier 1.5, 11 Tier 3) have two
contamination sources in A/B tests: code logic and LLM
non-determinism. Mitigation:

* Per-type prompts are pinned (no run-specific context leaks in).
* Caches are pinned per-project to a checked-in snapshot at
  `verusage/cache_snapshots/`.
* A/B runs use the same pinned cache for both sides; deltas in
  `pinned_hash`, `current_source_hash`, `pinned_hash_matches`,
  and per-name `gap_sources` tell you if the substrate changed.

### Sandbox + critic loops

Every LLM stage has both a static sandbox (rejects forbidden
constructs) *and* a critic LLM pass (semantic lints). The
combination is what makes LLM output trustworthy at the corpus
level: any path to acceptance requires both a clean sandbox scan
*and* Verus type-checking the output (Tier 1.5 V3 / V4, Tier 3
verus_pass, L4 verus --parse-only).

### Source mutations are isolated

The single-file driver always copies the source to a fresh temp
path before any rewriting, and writes the modified file to a
sibling ŌĆö the original is never touched. The cargo workspace
driver uses `inject_proof_fn` + `restore_file` in a try/finally
so an exception during the proof fn run still leaves the source
restored.

### Telemetry richness

Every result carries enough metadata to reproduce its bucket
verdict from the raw artifacts: `r0_z3` (ground truth), `permitted`
+ `permitted_reason` (Stage 12), `llm_assisted` (Stage 11),
`tier15` block (Stage 2), `n_schemas` / `n_rounds` / `verus_ms` /
`z3_ms` (cost). When triaging an unexpected bucket, the result
dict alone should be enough to decide whether to drill into
`injected.rs` or `llm_proof/`.

### Versioned data formats

`DetCheckSpec.to_dict` / `from_dict` are stable enough that
artifacts from 2026-04 still load under current code (with field
defaults for things added since). The Tier 1.5 cache and Tier 3
cache both carry an explicit `schema_version` and the loader
handles `v1 ŌåÆ v2` migration without erroring (`v1` entries simply
miss `shape_key` and don't participate in shape fallback).

## Appendix A ŌĆö schema kinds reference

| Kind | Param signature | Rendered assume | Use case |
|---|---|---|---|
| `SCALAR_EQ` | `g: bool, k: int` | `var == k` | Pin an integer to one value |
| `SCALAR_RANGE` | `g: bool, k_lo: int, k_hi: int` | `k_lo <= var && var <= k_hi` | Bisect integer range |
| `BOOL_EQ` | `g: bool, k: bool` | `var == k` | Pin a bool |
| `STR_EQ` | `g: bool` (per-literal id) | `var@ == "literal"@` | Pin a string to one of N candidates |
| `VARIANT_IS` | `g: bool` | `var is <Variant>` | Pick an Option/Result/enum variant |
| `ENUM_DISC_EQ` | `g: bool, k: int` | `var as int == k` | C-like enum |
| `SET_EMPTY` | `g: bool` | `var =~= Set::empty()` | Set is empty |
| `SET_LEN_GT` | `g: bool` | `var.len() > 0` | Set is non-empty |
| `SET_LEN_EQ` | `g: bool, k: int` | `var.len() == k` | Pin set length |
| `SET_LEN_RANGE` | `g: bool, k_lo, k_hi: int` | bisect length | Set length range |
| `SET_CONTAINS` | `g: bool, k: int` | `var.contains(k)` | Membership |
| `SEQ_LEN_EQ` | `g: bool, k: int` | `var.len() == k` | Pin seq length |
| `SEQ_LEN_RANGE` | `g: bool, k_lo, k_hi: int` | bisect length | Seq length range |
| `NOT_EQUAL_FN` | `g: bool` | `!det_<fn>_equal(r1, r2)` | The distinctness goal |

Adding a new kind:

1. Add a `SchemaKind` enum value in `schemas.py`.
2. Implement `enumerate` for it (which type tree shape generates it,
   with what parent guard chain).
3. Implement `render` (the `if g { assume(...); }` body it emits).
4. Implement `match_and_bind` on the matching `AssumePred` so the
   translator picks it up.
5. Add at least one schema-roundtrip test.

## Appendix B ŌĆö narrow strategy reference

| TypeKind | Strategy | Sub-strategy | Yields preds |
|---|---|---|---|
| INT / USIZE / ISIZE / U8..U64 / I8..I64 | `narrow_integer` | `_bisect_range`, then `_commit_eq` | `RangePred`, `EqPred` |
| BOOL | `narrow_bool` | enumerate | `BoolPred` |
| STR | `narrow_str` | enumerate K literals | `StrEqPred` |
| SET | `narrow_set` | empty ŌåÆ len ŌåÆ contains-each | `SetEmptyPred`, `LenEqPred`, `LenRangePred`, `SetContainsPred` |
| SEQ | `narrow_seq` | len ŌåÆ per-element recurse (capped at `MAX_SEQ_LEN`) | `LenEqPred`, `LenRangePred` + child preds |
| OPTION | `narrow_option` | `is Some`/`is None`, then recurse into `Some_0` | `VariantIsPred` + child |
| RESULT | `narrow_result` | `is Ok`/`is Err`, then recurse into payload | `VariantIsPred` + child |
| ENUM | `narrow_enum` | per-variant `VariantIsPred` + payload recurse | `VariantIsPred`, `DiscEqPred` + child |
| STRUCT | `narrow_struct` | recurse into each field | child preds |
| GHOST / TRACKED | unwrap | recurse into inner | inner preds |
| UNIT / OPAQUE | no-op | ŌĆö | ŌĆö |
| custom | `pass_untranslatable` | ŌĆö | ŌĆö |

Adding a new strategy:

1. Implement a function `narrow_<kind>(ty, var, node, ctx)` in
   `narrow.py`.
2. Register via `@register(TypeKind.<KIND>)`.
3. Add an `AssumePred` subclass if a new shape of assume is
   produced.
4. Wire pred ŌåÆ schema in `predicates.py` + `schemas.py`.
5. Add a self-test in `narrow.py` covering at least one bisect step.
