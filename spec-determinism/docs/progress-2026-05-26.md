## 2. Column reference

Every extracted target lands in exactly one column; per-project rows
sum to `total`. Mapping to `classify_ok` (in
`spec_determinism/classify.py`):

| column       | classify_ok bucket | underlying signal                                        |
|--------------|--------------------|----------------------------------------------------------|
| `total`      | (sum of below)     | det-check targets extracted for the project              |
| `complete`   | `complete`         | `status=ok` ∧ `r0_z3=unsat` ∧ no LLM assist              |
| `+LLM`       | `complete_llm`     | `status=ok` ∧ `r0_z3=unsat` ∧ `llm_assisted=True`        |
| `incomplete` | `incomplete`       | `status=ok` ∧ (`r0_z3=sat` ∨ (`r0_z3=unknown` ∧ `permitted=True`)) |
| `unknown`    | `ok_inconclusive`  | `status=ok` ∧ `r0_z3=unknown` ∧ `permitted=False`        |
| `crash`      | (status≠`ok`)      | `status=runner_crash` — driver hit the 300 s outer wall  |
| `verus_err`  | (status≠`ok`)      | `status=verus_error` — Verus refused to compile the file |

- `total` — one row per `(crate, function template)` the extractor
  found. For each row the pipeline synthesises a `det_<f>` proof fn
  (two parallel runs with arbitrary inputs + R0 equality check on the
  post-state) and hands the file to Verus + z3.
- `complete` — Verus compiled, z3 returned **R0=unsat**, no LLM proof
  block in the path. Spec's ensures uniquely pin the post-state.
- `+LLM` — same z3 verdict (R0=unsat) but unlocked by an LLM-authored
  proof block (Pattern A / E / C). Strict subset of "z3 proved
  determinism" — broken out so paper claims can separate "z3 alone"
  from "z3 + LLM-guided proof".
- `incomplete` — Verus compiled, but the spec admits multiple
  post-states. Two sub-cases:
  1. **R0=sat** — explicit z3 model where two runs reach different
     post-states (e.g. atmosphere `alloc_page_4k` returning any
     element of `free_pages_4k`).
  2. **R0=unknown ∧ `permitted=True`** — z3 surrendered but the spec
     explicitly uses one of the permissive markers
     (`permissive_or` via `|||`, or curated `REAL_SAT_MANUAL_FNS`
     allowlist). Classifier promotes these from `unknown` because the
     author already declared the spec to be non-deterministic.
- `unknown` — `r0_z3=unknown` without a permissive marker. z3
  surrendered, spec design provides no pardon. Could be either
  "stronger proof would make it complete" or "tighter SMT search
  would find a sat witness" — current evidence supports neither
  verdict. Roughly half are recoverable by `+LLM`; the rest time out
  or fail every LLM pattern.
- `crash` — per-target subprocess hit the 300 s outer wall before
  Verus finished. Distinct from the inner Verus `--timeout 120 s`
  (which surfaces as `r0_z3=unknown`). All 65 crashes are in
  atmosphere; root cause is z3 quantifier-instantiation explosion on
  the synthesised det check (see §4).
- `verus_err` — Verus's frontend refused to compile the file. Pure
  infra failure, not a determinism statement. The 2026-05-26 closeout
  drove this from 94 → 7 (the residual 7 are inherent storage source
  / vstd-version incompats, excluded from determinism numerator and
  denominator).

Pipeline-level source rewriters cleared **87 of 94** baseline
`verus_error` rows via three commits, lifting 47 to `complete`, 2 to
`incomplete`, and 36 to `unknown`:

| project    | baseline `verus_err` | post-closeout | newly `complete` | newly `incomplete` | newly `unknown` |
|------------|---------------------:|--------------:|-----------------:|-------------------:|----------------:|
| atmosphere |                   49 |             0 |               23 |                  0 |              24 |

| nrkernel   |                    2 |             0 |                1 |                  0 |               1 |
| **TOTAL**  |               **94** |         **7** |           **47** |              **2** |          **36** |

The 7 residual `verus_err` (all storage) are inherent source /
vstd-version incompats (`Box<S>: SpecEq<S>`, `iter.end` on
`VerusForLoopWrapper`) — not synthesiser bugs.

Three representative buckets, covering the spread of root causes:

- **43× unresolved `deps_hack::…`** — the sibling proc-macro crate (`PmSized` derive, `pmsized_primitive!` macro) doesn't exist in single-file mode. *Root cause: cross-crate dependency missing*; fixed by stripping the imports and emitting stub trait impls.
- **16× `Dereference this mutable reference`** — newer Verus rejects bare `self == old(self)` / `*p == old(*p)` in spec context. *Root cause: Verus version drift* since the corpus was authored; fixed by rewriting to `*self == *old(self)` / `final(p)`.
- **11× lost `&mut` on `Tracked(p): Tracked<&mut T>`** — the destructure pattern silently dropped the inner `&mut`. *Root cause: pipeline bug* in how the parameter pattern was normalised; fixed by preserving the inner annotation.

Per-bucket root-cause and fix tables, plus the pipeline files
touched, are in
[`progress-2026-05-26-details.md`](progress-2026-05-26-details.md) §1.

## 4. crash example — brief summary

`atmosphere::pagetable_map_4k_page` is the canonical `crash` case:
the synthesised det fn has 340 params + 566 `assume(...)`, and the
Verus → z3 query (`root.smt2`) is **29.5 MB / 1 481 (forall ...)
instances**. z3 falls into a matching loop on the
`get_pagetable_by_pcid(_).mapping_4k()` trigger heads and never
returns from a `(check-sat)` call before the 300 s outer wall fires.

The upstream function is `#[verifier::external_body]` with a
`unimplemented!()` body, meaning **atmosphere never asks z3 to
discharge the 76-line ensures** — they treat it as a spec axiom. Our
pipeline strips `external_body` to validate the spec, which is why
the crash surfaces. Across all 65 crashes: **37/65 (57 %) of crash
instances and 17/40 unique crash fns are `external_body` upstream.**

Full SMT-stage measurements, the source snippet, and the crash-fn
audit are in
[`progress-2026-05-26-details.md`](progress-2026-05-26-details.md) §2.

### 6.2 Other spec obligations (per [`spec-accountability/spec-obligations-overview-1.0.md`](../../spec-accountability/spec-obligations-overview-1.0.md))

Determinism (obligation **1.1**) is the line this repository has
operationalised end-to-end. The accountability overview defines five
further obligations under the same umbrella — a spec that the
verifier accepts is not necessarily a spec that *should* have been
written:

| # | Obligation | Group | One-line question |
|---|---|---|---|
| 1.1 | Determinism | Spec↔Impl | **Done** — does `S` shrink `A(S)` to one point under `eq_f`? |
| 1.2 | Attributive Non-Vacuity | Spec↔Impl | Does every clause of `S` actually contribute to constraining `A(S)`? |
| 2.1 | Intent Traceability & Drift | Spec↔Intent | Per method: do intent atoms and spec clauses align, and stay aligned under edits? |
| 2.2 | Compositional Sufficiency | Spec↔Intent | Does the top spec follow from the composition of callee specs along the call graph? |
| 2.3 | Correctness | Spec↔Intent | Does `A(S)` contain only intended impls? (deferred — structurally hardest) |
| 3.1 | Assumption Accountability | Meta | Is the trusted base behind every claim enumerated, audited, shared? |

Per the overview, the next two concrete moves are **1.2** and
**2.1**:

* **1.2 Attributive Non-Vacuity** — Direction A (entropy / #P
  counting) and Direction B (six clause-level structural sub-options)
  have both been explored without a satisfying primary direction;
  the next move is to converge on one.
* **2.1 Intent Traceability & Drift Sensitivity** — validate the
  per-method coverage matrix `M ∈ {0,1}^{|R|×|S|}` on a real Verus
  case study end-to-end: LLM-fill with grounded citations, human
  review, then run the static F1/F2/F3 scans (orphan requirement /
  orphan clause / mis-mapping) and the cross-version drift scans
  D1–D4.

2.2 and 3.1 stay as drafted plans, activating once 1.2 and 2.1
produce concrete artifacts to feed them. 2.3 is acknowledged and
held.
