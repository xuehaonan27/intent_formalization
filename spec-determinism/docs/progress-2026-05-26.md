## 2. Column reference

Every extracted target lands in exactly one column; per-project rows
sum to `total`. Mapping to `classify_ok` (in
`spec_determinism/classify.py`):

| column       | classify_ok bucket | underlying signal                                        |
|--------------|--------------------|----------------------------------------------------------|
| `total`      | (sum of below)     | det-check targets extracted for the project              |
| `complete`   | `complete`         | `status=ok` ‚àß `r0_z3=unsat` ‚àß no LLM assist              |
| `+LLM`       | `complete_llm`     | `status=ok` ‚àß `r0_z3=unsat` ‚àß `llm_assisted=True`        |
| `incomplete` | `incomplete`       | `status=ok` ‚àß (`r0_z3=sat` ‚à® (`r0_z3=unknown` ‚àß `permitted=True`)) |
| `unknown`    | `ok_inconclusive`  | `status=ok` ‚àß `r0_z3=unknown` ‚àß `permitted=False`        |
| `crash`      | (status‚âÝ`ok`)      | `status=runner_crash` ‚Äî driver hit the 300 s outer wall  |
| `verus_err`  | (status‚âÝ`ok`)      | `status=verus_error` ‚Äî Verus refused to compile the file |

- `total` ‚Äî one row per `(crate, function template)` the extractor
  found. For each row the pipeline synthesises a `det_<f>` proof fn
  (two parallel runs with arbitrary inputs + R0 equality check on the
  post-state) and hands the file to Verus + z3.
- `complete` ‚Äî Verus compiled, z3 returned **R0=unsat**, no LLM proof
  block in the path. Spec's ensures uniquely pin the post-state.
- `+LLM` ‚Äî same z3 verdict (R0=unsat) but unlocked by an LLM-authored
  proof block (Pattern A / E / C). Strict subset of "z3 proved
  determinism" ‚Äî broken out so paper claims can separate "z3 alone"
  from "z3 + LLM-guided proof".
- `incomplete` ‚Äî Verus compiled, but the spec admits multiple
  post-states. Two sub-cases:
  1. **R0=sat** ‚Äî explicit z3 model where two runs reach different
     post-states (e.g. atmosphere `alloc_page_4k` returning any
     element of `free_pages_4k`).
  2. **R0=unknown ‚àß `permitted=True`** ‚Äî z3 surrendered but the spec
     explicitly uses one of the permissive markers
     (`permissive_or` via `|||`, or curated `REAL_SAT_MANUAL_FNS`
     allowlist). Classifier promotes these from `unknown` because the
     author already declared the spec to be non-deterministic.
- `unknown` ‚Äî `r0_z3=unknown` without a permissive marker. z3
  surrendered, spec design provides no pardon. Could be either
  "stronger proof would make it complete" or "tighter SMT search
  would find a sat witness" ‚Äî current evidence supports neither
  verdict. Roughly half are recoverable by `+LLM`; the rest time out
  or fail every LLM pattern.
- `crash` ‚Äî per-target subprocess hit the 300 s outer wall before
  Verus finished. Distinct from the inner Verus `--timeout 120 s`
  (which surfaces as `r0_z3=unknown`). All 65 crashes are in
  atmosphere; root cause is z3 quantifier-instantiation explosion on
  the synthesised det check (see ¬ß4).
- `verus_err` ‚Äî Verus's frontend refused to compile the file. Pure
  infra failure, not a determinism statement. The 2026-05-26 closeout
  drove this from 94 ‚Üí 7 (the residual 7 are inherent storage source
  / vstd-version incompats, excluded from determinism numerator and
  denominator).

## 4. crash example ‚Äî `atmosphere::pagetable_map_4k_page`

Raw entry in `/tmp/corpus_baseline/atmosphere/full_run.json`:

```json
{
  "project":     "atmosphere",
  "file":        ".../verified/kernel/kernel__create_and_map_pages__impl0__alloc_and_map.rs",
  "function":    "pagetable_map_4k_page",
  "artifact_key": "atmosphere__verified__kernel__kernel__create_and_map_pages__impl0__alloc_and_map__pagetable_map_4k_page",
  "status":      "runner_crash",
  "error":       "subprocess wall timeout 300s",
  "wall_ms":     300035
}
```

Probed with `--keep-tmp` to recover the on-disk artefacts:

| where | value |
|---|---:|
| source ensures (lines 1741‚Äì1816) | **76 lines** |
| closed spec fns called from ensures | 8 (`wf`, `get_pagetable_by_pcid`, `get_pagetable_mapping_by_pcid`, `page_closure`, `mapping_4k/2m/1g`, `kernel_entries`) |
| synthesised det fn parameters | **340** |
| synthesised det fn signature size | 56 544 chars |
| synthesised det fn `assume(...)` count | **566** |
| synthesised det fn `forall` count | 12 |
| `reveal(...)` injected | **0** |
| Verus ‚Üí z3 query (`root.smt2`) | **29.5 MB / 85 292 lines** |
| ‚Ü≥ `(declare-fun ‚Ä¶)` | 851 |
| ‚Ü≥ `(assert ‚Ä¶)` | **2 501** |
| ‚Ü≥ `(forall ‚Ä¶)` instances | **1 481** |
| ‚Ü≥ `(check-sat)` | 3 |

A typical Verus query is in the tens-of-kB range with a few hundred
quantifier instances. This one is **30 MB / 1 481 forall** ‚Äî z3 falls
into a matching loop on the shared `get_pagetable_by_pcid(_).mapping_4k()`
trigger heads and never returns from one of the three `(check-sat)`
queries before the 300 s outer wall fires.

The Python driver (`verusage_run`) has **no child subprocess** during
synthesis ‚Äî gen_det's schema enumeration runs in-process and finishes
quickly; the wall is consumed entirely by the spawned `verus` ‚Üí `z3`
invocation. The 30 MB `root.smt2` is fully written to disk before
z3 hangs, confirming the bottleneck is downstream of synthesis.

### 4.1 Why atmosphere upstream "verifies" this function

The source declares:

```rust
#[verifier::external_body]
pub fn pagetable_map_4k_page(&mut self, ‚Ä¶) -> ‚Ä¶
    requires ‚Ä¶
    ensures ‚Ä¶   // 76 lines
{
    unimplemented!()
}
```

**Upstream atmosphere never proves this function.**
`#[verifier::external_body]` instructs Verus to trust the body and
expose only the ensures as an axiom to callers; the body is literally
`unimplemented!()`. The 76-line ensures was authored as a *spec
axiom*, not as something the atmosphere authors expected Verus to
discharge.

Our determinism pipeline strips `external_body` (it has to ‚Äî we
want to validate the *spec*, not skip it) and asks z3: "given these
ensures, is the output uniquely determined?" That requires unfolding
the full `wf()`-rooted closed-spec-fn cone ‚Äî which is exactly the
work atmosphere chose to externalise. The crash isn't a regression;
it surfaces the fact that the spec is too heavy for z3 to discharge
directly.

Auditing the 40 unique crash functions against the source: **17/40
(43 %) of unique fns** and **37/65 (57 %) of crash instances** carry
`#[verifier::external_body]` upstream ‚Äî i.e. the majority of the
crash *workload* comes from functions atmosphere itself never asked
z3 to verify. The remaining 23 unique fns (28 instances) have real
bodies, but they sit on the same `wf()`-axiom cone, so the
quantifier-instantiation pressure is the same.
