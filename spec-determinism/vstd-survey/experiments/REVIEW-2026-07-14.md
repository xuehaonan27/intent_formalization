# vstd determinism experiment review

## Scope

This experiment uses a matching Verus/vstd bundle:

- Verus version: `0.2026.05.17.e479cce`
- Verus commit: `e479cce36490b8fa4b0fd7755aa742aec354372c`
- vstd root: `/home/chentianyu/nanvix/toolchain/verus/vstd`

The current upstream survey is newer (`cf3b5c3`, 2026-07-13), but there is no
matching built Verus toolchain available yet. A matching source/binary pair is
used so version incompatibilities are not misclassified as specification
problems.

The matching snapshot contains 111 public exec definitions with explicit
postconditions:

- 34 free functions;
- 77 impl methods.

All targets use the repository's existing pipeline:

- `extract_spec`;
- `build_det_check_spec`;
- `ViewRegistry`;
- `enumerate_schemas` and `render_guarded_template`;
- `run_verus_file`;
- `build_schema_ctx` and `run_schema_search`;
- `classify_ok`.

## Overall results

| Scope | Targets | Complete | Inconclusive | Unsupported | Confirmed incomplete |
|---|---:|---:|---:|---:|---:|
| Public free functions | 34 | 27 | 5 | 2 | 0 |
| Public impl methods | 77 | 60 | 15 | 2 | 0 |
| **Total** | **111** | **87** | **20** | **4** | **0** |

`Complete` means `R0 = unsat` under a reviewed, non-trivial equal function.
`Inconclusive` means `R0 = unknown`; it is not counted as complete or
incomplete.

Six free raw-pointer functions initially received a trivial equality because
the default policy ignores raw-pointer identity. Re-running those targets with
`EqualPolicy(compare_raw_pointers=True)` proved all six complete; the table
uses the strict-pointer results.

No target produced a confirmed `sat` incompleteness witness.

## Inconclusive groups

### Explicitly or plausibly nondeterministic

- `float::float_cast`: the source describes `float_cast_spec` as a possibly
  non-deterministic Rust cast relation.
- `raw_ptr::allocate`: allocator address/provenance choice.
- `thread::spawn`: handle identity and predicate are not uniquely fixed.
- `thread::join`: successful values are constrained by the handle predicate,
  while the abstraction intentionally hides thread identity and runtime state.
- `cell::{empty, new}` and `simple_pptr::{empty, new}`: fresh identity/address
  allocation.
- `rwlock::{new, acquire_write, acquire_read, into_inner}`: opaque lock,
  handle, and permission identity.

### Equal-fn, view, or proof-context gaps

- `raw_ptr::ptr_ref2`: the ensures fix value, pointer address, and metadata,
  while equality still compares opaque `SharedReference` values.
- `thread::thread_id`: `IsThread::agrees` should relate calls on the same
  thread, but the generated proof does not instantiate the axiom.
- `atomic::{fetch_and, fetch_xor, fetch_or}`: permission post-state equality
  remains opaque to the type/narrowing pipeline.
- `InvCell::{new, replace, get}`: invariant-defined state is opaque and generic.
The seven original `StringHashMap` unknowns were equal-fn/view-bound false
unknowns. They now verify at R0 after the L3 view-bound fix documented in the
unknown audit.

## Unsupported targets

The four unsupported functions return mutable references:

- `array::ref_mut_array_unsizing_coercion`;
- `raw_ptr::ptr_mut_ref`;
- `cell::borrow_mut`;
- `simple_pptr::borrow_mut`.

For returned `&mut T`, ordinary occurrences denote the initial pointee and
`final(result)` denotes the final pointee. Current `gen_det` does not yet
substitute `old(result)`/`final(result)` correctly.

## Tooling findings and fixes

The experiment added or corrected:

1. Source-line-qualified extraction for same-named methods.
2. Artifact keys containing source lines.
3. ViewRegistry use in the vstd runner.
4. Snapshot compatibility for legacy `cell::PointsTo` and
   `simple_pptr::PointsTo` APIs.
5. Cross-module imports for hash and layout contracts.
6. Imported closed-spec handling without invalid `reveal` calls.
7. Detection of trivial `true` equal functions.
8. Strict raw-pointer equality experiments.
9. `PointsToRaw` opacity applying only to the outer type, not any tuple whose
   spelling contains `PointsToRaw`.
10. `Tracked<T>`/`Ghost<T>` using `@@` when the inner type has a spec view.
11. L3 generic view-bound checks requiring `View` only for parameters actually
    projected through `T::V`.
12. Registered View equality taking precedence over extensional equality on
    opaque wrapper structs.
