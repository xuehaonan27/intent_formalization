## Structured dependency graphs (`*_deps.json`)

For each project, `*_deps.json` holds a project-wide aggregate dep graph
generated by

```
python3 -m spec_determinism.type_registry deps <project>/verified \
    --aggregate-only --out <project>_deps.json
```

The aggregate collapses every short type name to a single node (cfg-dups,
multi-file copies merge), unions all referenced edges, and computes:

| key | meaning |
|---|---|
| `nodes` | every short type name appearing as a def or ref |
| `forward[T]` | direct types that `T` references |
| `forward_closure[T]` | transitive types reachable from `T` |
| `reverse[T]` | direct types that reference `T` |
| `reverse_closure[T]` | transitive types reaching `T` |
| `sccs` | strongly-connected components (Tarjan); non-trivial = cycles |
| `topological_order` | leaves-first order over the SCC condensation — ready-to-use bottom-up build order for witness gen |
| `classification[T]` | one of `struct` / `enum` / `union` / `alias` / `container` / `primitive` / `external` |
| `field_paths[QN]` | per-defined-type list of `{where, name, type_text, type_expr}` — `type_expr` is a recursive tree (`leaf` / `generic` / `ref` / `ptr` / `tuple` / `array` / `fn` / `dyn` / …) |

The collapsed-by-short-name node count is much smaller than the
`audit.json` per-file `total_types` because each file is self-contained
and the same `Page` / `Heap` / `Segment` / etc. is re-defined in many
files. The aggregate is what witness gen will actually consume — it
gives bottom-up build order over the *unique* types of the project.
