### 2 Evaluation extension: determinism feedback for spec generation

We plan to add an evaluation experiment showing that determinism feedback can enhance LLM-based spec generation.  The intended setting is
not a full spec-generation pipeline.  It is a small feedback loop:

1. Let an agent generate a candidate spec (can invoke verus until verified/no syntax error).
2. Run our determinism checker on the generated postcondition.
3. If the checker produces a witness (non-determinism), feed that
   diagnostic back to the agent and ask it to repair the spec.

This connects to the existing spec-generation / spec-evaluation landscape we
previously collected:

The pilot focuses on **Verus-SpecBench / SpecGym** because it is directly
NL→Spec and already gives a clean interface (`pre_spec`, `post_spec`) for
testing whether a generated postcondition leaves outputs underconstrained.

The current prototype uses a two-stage setup:

1. Generate and freeze a semantic output equivalence
   `specgen_output_equiv(in1, out1, out2)`.
2. Generate `pre_spec` / `post_spec` separately.
3. Generate the usual determinism proof obligation:

   ```rust
   proof fn det_post_spec(input: In, out1: Out, out2: Out)
       requires
           post_spec(input, out1),
           post_spec(input, out2),
       ensures
           specgen_output_equiv(input, out1, out2),
   { }
   ```

4. Run determinism checking with the frozen equivalence as the equal
   function.

### Preliminary observation

For many programming-contest tasks, the LLM naturally writes specs in
the form `result == judge(input)`, where `judge` is a spec-level
reimplementation of the intended algorithm.  For unique-output tasks
this often pins the output immediately; for multi-solution tasks, the
semantic equality must encode the "any valid answer" relation instead
of structural equality.

In a 10-case pilot with the two-stage setting, all 10 generated specs are
deterministic/complete, due to the characteristic of algorithm tasks.

##### Example: `verus-gym-1000-b`

The first clean example is `verus-gym-1000-b` (**Light It Up**).  The
problem gives a strictly increasing sequence of lamp switch times and
allows inserting at most one additional switch; the output is the
maximum total time the lamp can stay lit.  The equality is:

```rust
specgen_output_equiv(input, out1, out2) :=
    out1.max_lit_time == out2.max_lit_time
```

The generated postcondition has exactly the `result == judge(input)`
shape:

```rust
out.max_lit_time == specgen_best_lit_time_with_one_insert(...)
```

See the full generated judge in the
[1000-b case note](../spec-determinism/docs/specgen-det-feedback-case-1000b-2026-06-23.md).


## Future work (need discussion)

1. **How to design the spec-generation experiment.**
   The current SpecGym pilot suggests that many algorithmic tasks have
   the same pattern: a strong LLM writes a spec-level judge and then
   specifies `result == judge(input)`, so the generated postcondition is
   often already deterministic.  We need to decide whether the evaluation
   should:

   * keep SpecGym but use a weaker LLM / weaker prompt to create more
     under-specified candidates; or
   * switch to a benchmark where generated specs are more likely to
