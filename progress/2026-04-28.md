## 1. Scaling spec-determinism to VeruSage benchmarks

**Motivation:** Nanvix's verified modules are limited in number and use similar types (Bitmap, Slab, SortedVec), so the findings may not generalize. To evaluate whether our nondeterminism detection finds real-world spec gaps beyond Nanvix, I started running the tool on the VeruSage benchmark suite (720+ verified Verus files across multiple projects).

**Progress:** Ran the tool on the VeruSage benchmark suite — 1647 targets across 9 projects (atmosphere, ironkv, memory-allocator, nrkernel, vest, etc.). Results:

| Outcome | Count |
|---------|------:|
| Deterministic (PASS) | 878 |
| Nondeterministic witness found | 367 |
| Verus error (tool bug) | 401 |

Of the 367 witnesses, ~29 are **real incompleteness** (spec genuinely admits multiple post-states), ~280 are **false positives from over-comparison** (equal-fn compares private struct fields instead of the public view), and the rest are policy issues (nested Result/Option equality, lemma-shaped specs).

Of the 401 verus errors, the largest class (199) is a phantom-generics issue in the synthesized equal-fn — straightforward to fix.

**Issues encountered:** Documented in detail at [ISSUES.md](https://github.com/q5438722/intent_formalization/blob/main/spec-determinism/results-verusage/ISSUES.md). Key issues by priority:
1. **Equal-fn over-comparison** (~280 false witnesses + 13 verus errors): need view-aware equality (`a@ == b@` instead of field-by-field)
2. **Phantom generics in equal-fn** (199 verus errors): drop unused generic params from synthesized functions
3. **Ghost/Tracked type strategies** (29 real witnesses with unreadable output): need narrow strategies for `Tracked<T>`, `Ghost<T>`, `PointsTo<V>`
4. Several smaller issues (deref for `&mut`, variable name collision, parser errors)

## 2. Spec repair with overfitting prevention

**Motivation:** After detecting spec incompleteness, the next step is automated repair. A key challenge is preventing the LLM-based repair agent from overfitting — e.g., hardcoding witness constants or encoding implementation details into the spec.

I designed a repair pipeline with 5 quality criteria:
- **4 hard gates** (automated pass/fail): soundness (verus verification), determinism resolved (re-check), no witness constants (AST scan), vocabulary subset (symbol check)
- **1 soft ranking**: Minimum Description Length (MDL) over the repair's AST — combining tiered token cost (penalizing unfamiliar symbols) with structural cost (penalizing quantifier nesting, let bindings, case splits)

The pipeline generates N independent repair candidates via Copilot CLI agent sessions, filters through the hard gates, and ranks survivors by MDL score.

**Progress:** Built an end-to-end v0.1 repair pipeline (`spec-debug`):
- Classify nondeterminism witness assumes as **driving** vs. **collateral** (key insight: raw closed_count is misleading)
- Build prompt with witness + classified assumes + spec snippets → Copilot CLI → apply patch → regen → rerun → report
- Designed a multi-axis metrics framework (METRICS.md):
  - Hard gates: policy verdict, impl still verifies, no impl-side admissions, symbol table stable
  - Outcome ranking: lexicographic `(driving_closed_ratio, ¬new_witness_driving)`
  - Soft ranker: MDL (designed in REPAIR-CRITERIA.md, not yet implemented)
- Ran a 15-patch manual audit across 3 nanvix cases (bitmap::new, slab::from_raw_parts, kernel::from_raw_parts)
- Key finding: only 1 of 15 patches closed a **driving** assume. Most patches attacked collateral Err-side fields while the real gap was on the Ok side.
- Introduced **policy verdict** (A.1): counterfactual policy check before repair — if witness disappears under relaxed policy, redirect upstream instead of patching

Details: [STATUS.md](https://github.com/q5438722/intent_formalization/blob/main/spec-debug/STATUS.md)

## 3. Formalization of the framework

**Motivation:** Following Fan's suggestion, I formalized the overall approach.

**What was formalized:**
- Defined spec correctness, completeness, and determinism. Proved: under correctness, determinism ⟺ completeness (main theorem).
- Reduced the determinism check to a single Verus proof obligation (one `proof fn` with empty body).
- Defined a type-directed **witness lattice** (schema family over types: integers, booleans, variants, sets, struct fields via spec view) and a greedy **refinement search** algorithm on it. Proved soundness (every emitted witness is a real divergence) and saturation (search reaches the maximum witness expressible in the schema family).
- Designed the SMT oracle: one Verus compilation, then per-round assumption toggling via Z3's `check(*assumptions)` — sub-millisecond per round.
