# Weekly Progress — spec-determinism (week of 2026-05-12 → 05-19)

## 1. 整体漏斗 (overall funnel)

每个 verified fn 进入流程后会被分类到下列其中一个 bucket：

```
verified fn
    │
    ▼
extract (FunctionSpec)
    │
    ▼
gen_det (DetCheckSpec + equal_fn)
    │
    ▼
verus parse / typecheck ──► verus_error  (E0xxx 等编译/校验错误)
    │
    ▼
z3 check  !equal_fn(r1, r2, post1, post2)
    │
    ├──► unsat    ⇒ deterministic（已证明）
    ├──► sat      ⇒ real witness（真有不确定性）
    └──► unknown  ⇒ z3 决不下来
```

最后一份 ironkv baseline（上周交付时）：

| Bucket    | Count | 占比    |
| --------- | ----- | ----- |
| unsat (deterministic) |  94 | 43.9% |
| unknown (z3 undecided) |  76 | 35.5% |
| verus_error            |  44 | 20.6% |
| **total**              | **214** | 100% |

> 备注：当时这 76 被标成 "sat"，复查后确认其实是 `r0_z3=unknown`（z3 决不动），而不是真有 witness。

## 2. 处理 unknown 的分层策略 (Tier 模型)

unknown 的根因不止一种；我们按成本递增分了几个 tier：

| Tier  | 目标                                  | 手段                                                                                                   |
| ----- | ----------------------------------- | ---------------------------------------------------------------------------------------------------- |
| **T0** | 直接 z3                                 | 一发 `!equal_fn(...)`                                                                                  |
| **T1** | structural witness assumes           | Phase-3 schemas — 枚举 Set/Seq/Map 的 disjoint/identity 关系，给 z3 当 lemma                            |
| **T1.5** | **view-aware equal_fn** ⬅️ 本周主战场     | L1 prelude → L2 alias → L3 impl-scan → L4 LLM, 把 `equal(r1,r2)` 替换成 `r1@ == r2@`                  |
| **T2** | LLM proof loop                       | Codex/Claude 生成 `requires/ensures/proof block` 直到 verus 接受                                       |
| **T3** | LLM judge (待设计)                      | 把 spec ensures + equal_fn body + assumes 送 LLM 判 view-OK / 退回 struct-level                       |

每个 tier 都是 **incremental**：T0 unsat 就停；T1 假设跑过之后状态变成 `ok_witness`；T1.5 在 codegen 阶段就把 equal_fn 写对，等价于把 T0 直接送到目标。

### D. `_typeinfo_to_typeexpr` head normalization bug（今天）
在追 #1 时定位到一个 codegen bug：

- **症状**：`StrictlyOrderedMap<K>` 明明有 view (`type V = Map<K, ID>`)，但生成的 equal_fn 还是字段级 `(keys)@ == (keys)@) && ((vals)@ ...)`
- **根因**：extractor 把 raw 名字 `"StrictlyOrderedMap<K>"` 存进 `ty.name`；`_typeinfo_to_typeexpr` 直接把它当 `TypeExpr.head`；ViewRegistry 按 short name (`StrictlyOrderedMap`) 索引 → lookup miss → 静默 fallback 到字段级
- **修复**（5 行）：`if "<" in head: head = head.split("<", 1)[0].strip()`
- **smoke 验证**：`delegation_map_v__impl3__set::set` 从 `unknown` → **unsat (rounds=1, assumes=0)** ✅
  - 生成的 equal_fn：`(post1_self_).view() == (post2_self_).view()` （之前是 3 个字段 `&&` 链）

### E. View shim trait-bound 兼容性检查
- **回归发现**：rerun6 出 6 个 `verus_error`。`KeyIterator<K>` 的 L4-LLM 生成的 view shim 是 `impl<K: KeyTrait + VerusClone + View> View for KeyIterator<K>`（用了 `<K as View>::V`），但调用方 spec fn 只有 `<K: KeyTrait + VerusClone>` → E0277 `trait bound not satisfied`
- **修复**：`_shim_bounds_satisfied(shim_decl, caller_generics)` — 解析 shim 的 `impl<...>` bound 和 fn 的 `<...>` bound，bound 不是 subset 就拒绝 view-emission，回退字段级
- **结果**：rerun7 = 148 unsat / 66 unknown / **0 error** — 6 个 regression 全清除，4 个 view-win 保留

### 为什么 r8 还有这么多？

**opacity 只是 6 大类根因之一**。r8 的修复在 43 个目标上 fired，其中：
- 11 个原本已 unsat → 保持 unsat（**0 回归**）
