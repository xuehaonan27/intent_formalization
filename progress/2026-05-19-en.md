## 1. Overall funnel

Each verified fn enters the pipeline and lands in exactly one final
bucket:

```
verified fn
    │
    ▼
extract (FunctionSpec)
    │
    ▼
gen_det (DetCheckSpec + equal_fn)
    │
    ▼
verus parse / typecheck ──► verus_error  (E0xxx compile / verify error)
    │
    ▼
z3 check  !equal_fn(r1, r2, post1, post2)
    │
    ├──► unsat    ⇒ deterministic (proved)
    ├──► sat      ⇒ real witness (genuine non-determinism)
    └──► unknown  ⇒ z3 cannot decide
```

## 2. Layered strategy for `unknown` (Tier model)

`unknown` has multiple root causes. We address them in tiers of
increasing cost:

| Tier     | Goal                                       | Mechanism                                                                                                              |
| -------- | ------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------- |
| **T0**   | direct z3                                  | one-shot `!equal_fn(...)`                                                                                              |
| **T1**   | structural witness assumes                 | Phase-3 schemas — enumerate Set/Seq/Map disjoint/identity lemmas and feed them to z3                                   |
| **T1.5** | **view-aware equal_fn** ⬅️ main focus       | L1 prelude → L2 alias → L3 impl-scan → L4 LLM; rewrite `equal(r1, r2)` as `r1@ == r2@`                                 |
| **T2**   | LLM proof loop                             | Codex/Claude generates `requires/ensures/proof block` until verus accepts                                              |
| **T3**   | LLM judge (in design)                      | feed spec ensures + equal_fn body + assumes to an LLM to judge view-OK / fall back to struct-level                     |

Each tier is **incremental**: T0 stops if it returns unsat; T1
adds witness assumptions; T1.5 fixes the equal_fn at codegen time
(equivalent to handing T0 a tighter obligation from the start).

### D. `_typeinfo_to_typeexpr` head-normalisation bug (today)
While digging into #1 we found a codegen bug:

- **Symptom**: `StrictlyOrderedMap<K>` has a view (`type V = Map<K, ID>`),
  but the generated equal_fn falls back to field-level
  `(keys)@ == (keys)@) && ((vals)@ ...)`.
- **Cause**: the extractor stores the raw name
  `"StrictlyOrderedMap<K>"` in `ty.name`. `_typeinfo_to_typeexpr`
  uses that as `TypeExpr.head`. The ViewRegistry is keyed on short
  names (`StrictlyOrderedMap`), so the lookup misses and we
  silently fall back to field-level.
- **Fix** (5 lines): `if "<" in head: head = head.split("<", 1)[0].strip()`
- **Smoke**: `delegation_map_v__impl3__set::set` went from `unknown`
  → **unsat (rounds=1, assumes=0)** ✅
  Generated equal_fn: `(post1_self_).view() == (post2_self_).view()`
  (was a 3-field `&&` chain).

### E. View-shim trait-bound compatibility check
- **Regression spotted**: rerun6 surfaced 6 new `verus_error`s.
  The L4-LLM-generated view shim for `KeyIterator<K>` was
  `impl<K: KeyTrait + VerusClone + View> View for KeyIterator<K>`
  (uses `<K as View>::V`), but the calling spec fn only carries
  `<K: KeyTrait + VerusClone>` → E0277 *trait bound not satisfied*.
- **Fix**: `_shim_bounds_satisfied(shim_decl, caller_generics)`
  — parses both the shim's `impl<...>` bounds and the caller's
  `<...>` bounds; if the shim's bounds aren't a subset, reject
  view-emission and fall back to field-level.
- **Result**: rerun7 = 148 unsat / 66 unknown / **0 error** —
  6 regressions cleared, all 4 view-wins retained.

### Why are there still so many in r8?

**Opacity is only one of six root causes**. r8's fix fired on 43
targets; of those:
- 11 were already `unsat` → stay `unsat` (**0 regressions**)
- 32 were `unknown` → **z3 still returns unknown**

Of those 32, ~24 fall into categories 4–5 (quantifier wall /
vec-search). For these, exposing the body to z3 doesn't help — the
problem is not "z3 can't see the body", it's "even with the body in
view, z3 can't trigger the right instantiation". These need an LLM
to write `assert by { ... }` lemma hints to break through.

The 9 manually-identified REAL_SAT cases (5 spec fns) are catalogued
in [`docs/ironkv-real-sat-cases-2026-05-19.en.md`](../spec-determinism/docs/ironkv-real-sat-cases-2026-05-19.en.md).
